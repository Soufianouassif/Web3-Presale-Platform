أريد تطوير نظام الجلسات (Session System) في موقعي ليصبح أكثر تطوراً، أكثر أماناً، وأكثر احترافية، بمستوى Production حقيقي.

مهم جداً:
- لا أريد تعديلات سطحية
- لا أريد مجرد تغيير بسيط في cookie
- أريد مراجعة شاملة لنظام الجلسات والمصادقة والحماية
- لا تكسر تسجيل الدخول الحالي
- لا تكسر لوحة المستخدم أو لوحة الأدمن
- حسّن النظام بشكل نظيف ومنظم

الهدف
━━━━━━━━━━━━━━━━━━

أريد نظام جلسات قوي وآمن يدعم:

1. حماية قوية ضد session hijacking
2. حماية ضد session fixation
3. انتهاء جلسة صحيح وآمن
4. logout حقيقي ينهي الجلسة بالكامل
5. تمييز واضح بين user session و admin session
6. حماية الصفحات والـ APIs حسب الصلاحيات
7. إدارة session lifecycle بشكل صحيح
8. دعم أفضل للأمان في production

━━━━━━━━━━━━━━━━━━
المرحلة 1 — فحص النظام الحالي
━━━━━━━━━━━━━━━━━━

افحص نظام الجلسات الحالي بالكامل.

تحقق من:
- كيف يتم إنشاء session
- أين يتم حفظ session
- ما هو session store المستخدم
- هل هو memory store فقط؟
- هل cookie settings آمنة؟
- هل session id يعاد توليده بعد login؟
- هل logout يدمر الجلسة فعلاً؟
- هل توجد routes محمية فعلاً؟
- هل توجد admin routes غير محمية جيداً؟
- هل الجلسة تستمر أكثر من اللازم؟
- هل كل المستخدمين يخضعون لنفس نوع الجلسة بدون فصل صلاحيات؟

أريد منك تحديد نقاط الضعف بدقة قبل الإصلاح.

━━━━━━━━━━━━━━━━━━
المرحلة 2 — تحسين أمان Cookie و Session
━━━━━━━━━━━━━━━━━━

أريد إعدادات آمنة ومحترفة للجلسة:

- httpOnly: true
- secure: true في production
- sameSite: 'lax' أو 'strict' حسب الحاجة
- maxAge مناسب
- لا تستخدم expires ثابتة محسوبة عند بدء السيرفر
- اسم cookie واضح وغير افتراضي إن لزم
- حماية cookie من الوصول عبر JavaScript
- راجع trust proxy إذا كان المشروع خلف proxy أو Vercel / Replit / Nginx

إذا كانت الإعدادات الحالية ضعيفة:
- أصلحها

المرحلة 3 — Session Fixation Protection
━━━━━━━━━━━━━━━━━━

أريد حماية كاملة ضد Session Fixation.

تحقق من:
- هل req.session.regenerate() يُستخدم بعد login؟
- هل يتم توليد session جديدة بعد المصادقة؟
- هل session القديمة تُستبدل فعلاً؟
- هل OAuth login و wallet login يطبقان نفس الحماية؟

إذا لم يكن هذا مطبقاً:
- أضفه بشكل صحيح

━━━━━━━━━━━━━━━━━━
المرحلة 4 — Logout و Session Destruction
━━━━━━━━━━━━━━━━━━

أريد logout احترافي وآمن.

تحقق من:
- هل logout يدمر session من السيرفر؟
- هل cookie تُحذف فعلاً؟
- هل تبقى الجلسة صالحة بعد logout؟
- هل يمكن الرجوع بزر back والوصول لصفحات محمية؟
- هل هناك cache أو state محلية توهم أن المستخدم ما زال مسجلاً؟

أريد:
- destroy session
- clear cookie
- منع بقاء صلاحية الجلسة بعد logout
- redirect أو refresh مناسب بعد logout

━━━━━━━━━━━━━━━━━━
المرحلة 5 — حماية الصلاحيات Roles & Access Control
━━━━━━━━━━━━━━━━━━

أريد فصل الصلاحيات بشكل واضح وآمن.

أضف أو حسّن:

1. Middleware للمستخدم العادي
- يتأكد أن المستخدم مسجل الدخول

2. Middleware للأدمن
- يتأكد أن المستخدم أدمن فعلاً
- يمنع أي user عادي من الوصول إلى admin pages أو admin APIs

3. حماية API Routes
- لا يكفي حماية الواجهة فقط
- يجب حماية backend APIs أيضاً

4. التحقق من session userId / role / wallet / account status
- لا تعتمد فقط على وجود session
- تحقق من role والصلاحيات

━━━━━━━━━━━━━━━━━━
المرحلة 6 — Session Store احترافي
━━━━━━━━━━━━━━━━━━

تحقق من مكان تخزين الجلسات.

إذا كان النظام يستخدم MemoryStore فقط:
- هذا غير مناسب للإنتاج
- استبدله بحل أفضل إن أمكن مثل:
  Redis session store
  أو database-backed session store
  أو حل احترافي مناسب للبنية الحالية

أريد session store يتحمل:
- إعادة تشغيل السيرفر
- تعدد المستخدمين
- عدم ضياع الجلسات بسهولة
- قابلية التوسع

إذا تعذر تطبيق Redis حالياً:
- نظم الكود بحيث يمكن إضافة store احترافي لاحقاً بسهولة

━━━━━━━━━━━━━━━━━━
المرحلة 7 — Session Lifetime و Idle Timeout
━━━━━━━━━━━━━━━━━━

أريد تحسين دورة حياة الجلسة.

أضف أو راجع:
- مدة الجلسة الكلية
- idle timeout عند عدم النشاط
- تجديد الجلسة عند النشاط الحقيقي
- منع بقاء جلسات طويلة بشكل خطير
- إذا كانت هناك صفحة admin، اجعل أمانها أعلى ومدة جلستها أقصر إن لزم

أريد منطق واضح:
- متى تنتهي الجلسة؟
- متى يتم تجديدها؟
- كيف يتصرف النظام إذا انتهت؟

━━━━━━━━━━━━━━━━━━
المرحلة 8 — حماية إضافية للأدمن
━━━━━━━━━━━━━━━━━━

لوحة الأدمن تحتاج حماية أقوى.

أريد:
- حماية قوية على جميع admin routes
- التحقق من role في backend
- logging لمحاولات الوصول المرفوضة
- optional re-check للهوية عند actions حساسة مثل:
  pause sale
  resume sale
  config update
  referral reward management
  price sync
- إذا كان ممكناً، أضف طبقة حماية إضافية للأدمن مثل:
  require recent login
  أو خطوة تحقق إضافية قبل العمليات الحساسة

━━━━━━━━━━━━━━━━━━
المرحلة 9 — Session Debugging و Logging
━━━━━━━━━━━━━━━━━━

أريد logging آمن ومفيد لتشخيص مشاكل الجلسات، بدون تسريب معلومات حساسة.

سجل:
- login success
- login failure
- session regenerated
- logout
- unauthorized access attempts
- forbidden admin access attempts
- session expired events

لكن:
- لا تسجل session secrets
- لا تسجل cookies كاملة
- لا تسجل private data بشكل خطر

المرحلة 10 — Frontend Session Handling
━━━━━━━━━━━━━━━━━━

افحص الواجهة أيضاً:

- هل frontend يتحقق من session فعلاً؟
- هل session state تتحدث بعد login/logout؟
- هل UI تعرض بيانات مستخدم قديم بعد logout؟
- هل protected pages يمكن فتحها بدون تحقق حقيقي؟
- هل يوجد stale state أو cached user info؟

أصلح:
- تحديث حالة المستخدم بعد login
- تحديثها بعد logout
- منع الواجهة من عرض بيانات قديمة
- redirect صحيح عند انتهاء الجلسة

المرحلة 11 — ما أريده عملياً
━━━━━━━━━━━━━━━━━━

أريد نظام جلسات يدعم على الأقل:

- login آمن
- logout آمن
- session regeneration
- protected routes
- role-based access
- secure cookies
- production-safe settings
- session store أفضل من memory إذا أمكن
- session expiry منطقية
- admin protection أقوى

المرحلة 12 — التقرير النهائي
━━━━━━━━━━━━━━━━━━

بعد الفحص والإصلاح، أعطني تقريراً واضحاً جداً:

1. ما هي مشاكل نظام الجلسات الحالية
2. ما مستوى خطورة كل مشكلة
3. ما الذي تم إصلاحه فعلياً
4. ما الملفات التي تم تعديلها
5. هل كان هناك memory store فقط؟
6. هل تم تطبيق session regeneration؟
7. هل logout أصبح يدمر الجلسة فعلاً؟
8. هل admin routes أصبحت محمية فعلاً؟
9. كيف أختبر النظام بنفسي خطوة بخطوة

━━━━━━━━━━━━━━━━━━
قواعد صارمة
━━━━━━━━━━━━━━━━━━

- لا تفترض أن النظام آمن
- لا تكتفِ بتعديلات شكلية
- لا تقل "تم الإصلاح" إلا بعد مراجعة شاملة
- لا تكسر منطق الموقع الحالي
- لا تكسر تسجيل الدخول أو الشراء أو لوحة الأدمن
- أصلح root cause وليس الأعراض
- اجعل الكود منظم وقابل للتوسع
