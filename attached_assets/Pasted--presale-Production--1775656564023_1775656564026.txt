قم بتحويل مشروع الـ presale الحالي إلى مستوى Production حقيقي مع إضافة طبقات حماية قوية ضد السبام، البوتات، والشراء الوهمي.

مهم جداً:
- لا تغيّر منطق المشروع الأساسي
- لا تكسر الواجهة الحالية
- لا تحذف أي ميزة موجودة
- قم فقط بإضافة الحماية والتحسينات الأمنية والإنتاجية بشكل نظيف ومنظم

━━━━━━━━━━━━━━━━━━
المرحلة 1 — حماية Presale API
━━━━━━━━━━━━━━━━━━

افحص جميع endpoints المتعلقة بالشراء، الإحالة، الأسعار، والـ admin.

أضف الحماية التالية:

1. Rate Limiting قوي ومناسب لكل endpoint
- endpoints العامة: 60 طلب / دقيقة
- endpoints الحساسة مثل الشراء والإحالة: 10 طلبات / دقيقة
- admin endpoints: 30 طلب / 5 دقائق
- cron endpoints: 3 طلبات / 5 دقائق

2. Anti-Bot Protection
- أضف فحص User-Agent
- امنع الطلبات الفارغة أو المشبوهة
- أضف basic bot detection للطلبات المتكررة جداً من نفس IP
- إن أمكن أضف حماية CAPTCHA-ready architecture بدون كسر الكود الحالي
- جهز مكان واضح لإضافة Cloudflare Turnstile أو Google reCAPTCHA لاحقاً

3. Input Validation
- تحقق من جميع body / params / query باستخدام validation صارم
- wallet addresses
- purchaseId
- referralCode
- username
- page / limit
- amount values
- أي input من المستخدم يجب أن يتم تنظيفه والتحقق منه

━━━━━━━━━━━━━━━━━━
المرحلة 2 — حماية عمليات الشراء
━━━━━━━━━━━━━━━━━━

تحقق من منطق الشراء بالكامل.

1. لا تثق بأي قيمة قادمة من العميل:
- amountUsd
- amountTokens
- token price
- referral reward
- stage data

2. اجعل السيرفر هو المصدر الحقيقي للحسابات:
- احسب amountTokens في السيرفر
- احسب amountUsd في السيرفر أو من مصدر موثوق
- لا تعتمد على frontend في القيم المالية

3. Duplicate Purchase Protection
- امنع تسجيل نفس transaction أكثر من مرة
- امنع replay attacks
- تأكد أن كل tx hash أو signature تُسجل مرة واحدة فقط

4. Wallet Integrity
- تأكد أن المحفظة المُرسلة متوافقة مع العملية
- امنع تسجيل purchase باسم محفظة مختلفة عن المنفذ الحقيقي إن كان ذلك ممكناً منطقياً

━━━━━━━━━━━━━━━━━━
المرحلة 3 — حماية نظام الإحالة Referral
━━━━━━━━━━━━━━━━━━

افحص نظام referral بالكامل.

أضف ما يلي:
- التحقق أن purchaseId موجود فعلاً
- التحقق أن عملية الشراء ناجحة فعلاً
- التحقق أن الإحالة لا تُكرر
- منع self-referral
- منع referral loops
- منع إنشاء مكافآت من purchases مزيفة
- اجعل مكافآت referral تُحسب فقط من بيانات موثوقة من السيرفر أو قاعدة البيانات

إذا كان هناك أي خلل منطقي في referral rewards:
- أصلحه مباشرة
- حافظ على نفس السلوك المتوقع للمستخدم

━━━━━━━━━━━━━━━━━━
المرحلة 4 — أمان الجلسات والمصادقة
━━━━━━━━━━━━━━━━━━

تحقق من auth system بالكامل.

يجب:
- استخدام req.session.regenerate() بعد تسجيل الدخول
- منع Session Fixation
- تفعيل:
  httpOnly: true
  sameSite: 'lax' أو 'strict'
  secure: true في production
- استخدام maxAge بدل expires الثابت
- تنظيف الجلسة عند logout بشكل صحيح
- التأكد أن الجلسة لا تبقى صالحة بعد تسجيل الخروج
- حماية admin routes بمصادقة حقيقية وصارمة

━━━━━━━━━━━━━━━━━━
المرحلة 5 — حماية Admin Panel
━━━━━━━━━━━━━━━━━━

افحص جميع admin APIs وصفحات الإدارة.

أضف:
- middleware واضح للتحقق من admin
- منع الوصول غير المصرح به
- تقييد page و limit
- حماية أي actions حساسة مثل:
  pause sale
  resume sale
  announcements
  config updates
  price sync
- أضف audit logging لكل admin action مهم:
  من قام بالفعل
  متى
  ما الذي تم تغييره

━━━━━━━━━━━━━━━━━━
المرحلة 6 — Environment Variables و Secrets
━━━━━━━━━━━━━━━━━━

افحص جميع الأسرار والمتغيرات البيئية.

تأكد من:
- عدم وجود أي secrets مكتوبة مباشرة في الكود
- private keys فقط عبر environment variables
- CRON_SECRET موجود ومستخدم
- SESSION_SECRET قوي
- أي API keys ليست hardcoded
- أضف فحص startup validation:
  إذا كانت متغيرات حساسة ناقصة، يجب أن يفشل السيرفر برسالة واضحة وآمنة

━━━━━━━━━━━━━━━━━━
المرحلة 7 — Blockchain Safety
━━━━━━━━━━━━━━━━━━

افحص الربط مع البلوكتشين.

تأكد من:
- تطابق Program ID بين frontend و backend و env
- تطابق IDL مع البرنامج الحقيقي
- التحقق من network الصحيحة
- عدم تنفيذ أي عملية حساسة بدون authorization
- أي endpoint يتعامل مع البلوكتشين يجب أن يكون محمي ومحدود

إذا وُجد mismatch:
- أصلحه
- أعط تقرير واضح بما تم تعديله

━━━━━━━━━━━━━━━━━━
المرحلة 8 — Logging و Monitoring
━━━━━━━━━━━━━━━━━━

أضف logging منظم وآمن:

- سجل الأخطاء المهمة
- سجل محاولات الوصول المرفوضة
- سجل محاولات admin
- سجل rate limit hits
- لا تسجل secrets
- لا تسجل private keys
- لا تسجل session cookies

اجعل الـ logs مفيدة للتشخيص بدون تسريب معلومات حساسة

━━━━━━━━━━━━━━━━━━
المرحلة 9 — Production Hardening
━━━━━━━━━━━━━━━━━━

اجعل السيرفر جاهز للإنتاج:

- أضف Helmet إذا لم يكن موجوداً
- أضف CORS مضبوطاً بشكل صحيح
- أضف body size limits
- امنع headers غير الضرورية
- أضف trust proxy إذا كان مناسباً
- حسّن error handling:
  لا تُرجع stack traces للمستخدم
  أظهر رسائل آمنة فقط
- أضف centralized error handler إن لم يكن موجوداً

━━━━━━━━━━━━━━━━━━
المرحلة 10 — التقرير النهائي
━━━━━━━━━━━━━━━━━━

بعد الفحص والإصلاح:
أعط تقريراً واضحاً جداً يحتوي على:

1. قائمة الثغرات التي تم العثور عليها
2. مستوى الخطورة لكل واحدة
3. هل تم إصلاحها أم لا
4. اسم الملف والسطر أو القسم المعدل
5. ملخص قصير لكل تعديل
6. أي مشاكل لم يمكن إصلاحها بالكامل مع السبب

━━━━━━━━━━━━━━━━━━
قواعد صارمة
━━━━━━━━━━━━━━━━━━

- لا تعطِ نصائح عامة فقط
- عدّل الكود فعلياً عند الحاجة
- لا تكسر الـ frontend
- لا تكسر wallet flow
- لا تكسر purchase flow
- لا تغيّر UI إلا إذا كان ذلك ضرورياً للحماية
- اجعل جميع التعديلات نظيفة ومنظمة وجاهزة للإنتاج
- إذا أضفت middleware أو helpers جديدة، نظّمها بشكل احترافي
