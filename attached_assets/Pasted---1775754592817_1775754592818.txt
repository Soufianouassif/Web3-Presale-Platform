أريد الآن مرحلة حاسمة ونهائية لنظام الإحالة. لا أريد مزيداً من الافتراضات أو الرسائل العامة مثل "تم الإصلاح". أريد إثباتاً عملياً في الكود وفي الواجهة وفي قاعدة البيانات أن النظام يعمل end-to-end بشكل ديناميكي وحقيقي.

المشكلة الحالية:
- سبق إصلاح بعض الأخطاء في frontend وbackend
- لكن ما زلت غير واثق هل لوحة المحيل ولوحة الأدمن تعرضان بيانات حقيقية أم static/mock data
- وأريد أيضاً تحسين دقة reward calculation وعدم الاعتماد على fallback غير دقيق إلا عند الضرورة القصوى

الهدف النهائي
━━━━━━━━━━━━━━━━━━

أريد منك تنفيذ فحص وإصلاح نهائي لنظام الإحالة بحيث يصبح:

1. Dynamic بالكامل
2. مربوط فعلياً بقاعدة البيانات
3. يعرض بيانات صحيحة في لوحة المحيل
4. يعرض بيانات صحيحة في لوحة الأدمن
5. يحسب المكافآت بدقة ومن مصدر موثوق
6. لا يحتوي على mock data أو placeholder values مضللة
7. قابل للاختبار بوضوح من أول click على referral link إلى ظهور البيانات في dashboards

المرحلة 1 — إثبات ديناميكية الواجهة
━━━━━━━━━━━━━━━━━━

افحص جميع الملفات المتعلقة بـ:
- referral dashboard
- investor dashboard
- admin referral page
- referral stats cards
- referral tables
- reward summary
- latest referred purchases
- admin referral overview

تحقق بدقة من الأمور التالية:

- هل يوجد أي mockData أو dummyData أو placeholderData؟
- هل توجد arrays مكتوبة يدوياً داخل components؟
- هل توجد أرقام ثابتة hardcoded؟
- هل توجد useState بقيم افتراضية لا يتم استبدالها ببيانات حقيقية؟
- هل الصفحة تعرض demo rows أو fake stats؟
- هل توجد props ثابتة تُغذي cards بدل بيانات backend؟

إذا وجدت أي شيء static:
- احذفه أو استبدله مباشرة بربط حقيقي مع API/DB
- إذا لم توجد بيانات، اعرض empty state حقيقية وواضحة بدل قيم مزيفة

المرحلة 2 — تحديد source الحقيقي لكل صفحة
━━━━━━━━━━━━━━━━━━

أريد تحديداً واضحاً جداً لكل صفحة:

1. لوحة المحيل:
- ما endpoint الذي تستعمله؟
- ما query/service الذي يجلب:
  total referrals
  total rewards
  total referred purchases
  referral history
  latest referred buyers

2. لوحة الأدمن:
- ما endpoint الذي تستعمله؟
- ما query/service الذي يجلب:
  all referrals
  reward totals
  buyer wallet
  referrer wallet
  purchase amount
  reward amount
  status
  createdAt

أريدك أن تتحقق أن هذه الصفحات مربوطة فعلاً بـ backend الحقيقي وليس state محلية أو data ثابتة.

━━━━━━━━━━━━━━━━━━
المرحلة 3 — Logging حقيقي في القراءة والعرض
━━━━━━━━━━━━━━━━━━

أضف logging واضح ومؤقت في:

- endpoint لوحة المحيل
- endpoint لوحة الأدمن
- service/query التي تجلب referral stats
- service/query التي تجلب referral history
- service/query التي تجلب reward totals

أريد logs تبين:
- أي user / wallet يطلب البيانات
- عدد السجلات المرجعة
- total rewards
- total referrals
- source query result
- إذا كانت النتيجة فارغة، لماذا

لا تسجل بيانات حساسة بشكل خطر، لكن اجعل logs كافية للتأكد أن القراءة من DB تعمل فعلاً.

المرحلة 4 — مراجعة قاعدة البيانات والعلاقات
━━━━━━━━━━━━━━━━━━

افحص الجداول المرتبطة:
- purchases
- referrals
- referral_rewards
- users / investors / wallets
- أي جداول وسيطة ذات علاقة

تحقق بدقة من:
- هل purchase تُحفظ فعلاً؟
- هل تُربط بالمحيل الصحيح؟
- هل referral record يُنشأ؟
- هل reward record يُنشأ؟
- هل هناك mismatch بين أسماء الحقول في الكود والـ DB؟
- هل هناك joins خاطئة؟
- هل هناك فلترة خاطئة تجعل dashboards ترجع فارغة؟
- هل status أو conditions تمنع ظهور البيانات؟

إذا وجدت mismatch:
- أصلحه مباشرة

━━━━━━━━━━━━━━━━━━
المرحلة 5 — تحسين دقة احتساب المكافأة
━━━━━━━━━━━━━━━━━━

أريد مراجعة منطق reward calculation بالكامل.

المهم:
- لا تعتمد على solPrice = 0
- لا تعتمد على fallback ثابت مثل 150 إلا كحل مؤقت جداً ومعلّق بوضوح
- الأفضل:
  1. جلب السعر من backend أو مصدر موثوق
  2. أو منع tracking إذا السعر غير جاهز
  3. أو استخدام trusted server-side calculation
  4. أو ربط الحساب بقيمة purchase موثوقة محفوظة من العملية

تحقق من:
- كيف يتم حساب amountUsd
- كيف يتم حساب amountTokens
- كيف يتم حساب rewardAmount
- هل الحساب يحصل في frontend أم backend؟
- ما هو source of truth الحقيقي؟

أريد أن يصبح الحساب:
- دقيقاً
- منطقياً
- قابلاً للتدقيق
- غير معتمد على قيم سهلة الكسر من جهة العميل

إذا كان من الضروري مؤقتاً استعمال fallback:
- اجعله واضحاً في الكود
- أضف warning log
- وبيّن أنه temporary فقط

━━━━━━━━━━━━━━━━━━
المرحلة 6 — إعادة التحديث بعد الشراء
━━━━━━━━━━━━━━━━━━

أريد التأكد من أن dashboards تتحدث فعلياً بعد شراء جديد.

تحقق من:
- هل يوجد refetch بعد نجاح الشراء؟
- هل توجد cache invalidation إذا كان React Query أو SWR مستخدماً؟
- هل الواجهة تحتاج refresh يدوي؟
- هل state تبقى قديمة؟
- هل user يرى نفس الأرقام حتى بعد نجاح العملية؟

أصلح ذلك حتى:
- بعد شراء جديد عبر referral
- تظهر البيانات الصحيحة في dashboard
- وتظهر كذلك في admin
- إما مباشرة أو بعد refresh واضح ومفهوم

━━━━━━━━━━━━━━━━━━
المرحلة 7 — اختبار End-to-End حقيقي
━━━━━━━━━━━━━━━━━━

أريد اختباراً حقيقياً لهذه السلسلة كاملة:

1. فتح الموقع عبر referral link
2. التقاط ref code من URL
3. حفظه بشكل صحيح
4. تنفيذ شراء جديد
5. إرسال referral code مع purchase request
6. حفظ purchase في DB
7. إنشاء referral record
8. إنشاء reward record
9. تحديث لوحة المحيل
10. تحديث لوحة الأدمن

إذا تعذر تنفيذ الاختبار تلقائياً:
- أضف logs مفصلة
- أو صفحة debug مؤقتة
- أو endpoint debug آمن في بيئة التطوير فقط
بحيث يمكن رؤية كل مرحلة بوضوح

المرحلة 8 — إزالة أي وهم بصري في النظام
━━━━━━━━━━━━━━━━━━

إذا كانت أي صفحة في production تعرض:
- fake numbers
- sample rows
- dummy referrals
- placeholder rewards
- demo cards

فأزلها فوراً أو استبدلها بـ:
- بيانات حقيقية من DB
أو
- empty states واضحة مثل:
  No referrals yet
  No rewards yet
  No referral purchases yet

لا أريد أن يبدو النظام شغالاً بينما هو مجرد واجهة تجريبية.

المرحلة 9 — التقرير النهائي الصارم
━━━━━━━━━━━━━━━━━━

في النهاية أعطني تقريراً واضحاً جداً ومفصلاً:

1. هل كانت لوحة المحيل static أم dynamic؟
2. هل كانت لوحة الأدمن static أم dynamic؟
3. ما الملفات التي كانت تحتوي على بيانات ثابتة أو مضللة؟
4. ما الملفات التي تم تعديلها؟
5. ما endpoints الحقيقية التي تستعملها كل صفحة الآن؟
6. هل dashboards أصبحت تجلب البيانات من قاعدة البيانات فعلاً؟
7. هل reward calculation أصبح أدق؟
8. ما الذي ما زال مؤقتاً أو يحتاج تحسين لاحق؟
9. كيف أختبر بنفسي خطوة بخطوة بعد الإصلاح؟

━━━━━━━━━━━━━━━━━━
قواعد صارمة جداً
━━━━━━━━━━━━━━━━━━

- لا تقل "تم الإصلاح" إلا بعد تتبع القراءة والعرض من قاعدة البيانات إلى الواجهة
- لا تفترض أن أي dashboard ديناميكي
- لا تترك أي mock data في صفحات الإحالة
- لا تعتمد على fallback مالي ضعيف كحل نهائي
- أصلح root cause وليس الأعراض
- لا تكسر purchase flow أو admin flow أو investor dashboard
- إذا كانت هناك أكثر من مشكلة، أصلحها كلها
- أريد نتيجة production-like وليست patch سطحية
