أعتقد أن مشكلة نظام الإحالة في موقعي ليست فقط في الحساب أو الحفظ، بل أيضاً في أن الواجهة نفسها تبدو static وليست dynamic.

المشكلة التي ألاحظها:
- بيانات الإحالة في لوحة المحيل لا تتحدث
- المكافآت لا تتغير
- لوحة الأدمن لا تعرض بيانات حقيقية
- أشعر أن الموقع يعرض بيانات ثابتة أو mock data أو placeholder data بدل البيانات الحقيقية من قاعدة البيانات

أريد منك فحصاً صارماً وحقيقياً للتأكد هل صفحات الإحالة والداشبورد ديناميكية فعلاً أم مجرد واجهات ثابتة.

أريد منك التحقق من هذا المسار بالكامل:

1. هل بيانات الإحالة تُحفظ فعلاً في قاعدة البيانات؟
2. هل توجد API endpoints حقيقية تعيد هذه البيانات؟
3. هل الواجهة تستدعي هذه endpoints فعلاً؟
4. هل الواجهة تعرض البيانات الحقيقية أم mock/static data؟
5. هل البيانات يتم تحديثها بعد الشراء أو بعد refresh؟
6. هل لوحة المحيل ولوحة الأدمن مربوطتان بنفس source الصحيح؟

المرحلة 1 — فحص الواجهة
━━━━━━━━━━━━━━━━━━

افحص جميع الملفات الخاصة بـ:
- referrer dashboard
- referral stats cards
- referral history table
- admin referral page
- admin referral stats
- أي component يعرض:
  total referrals
  total rewards
  referral purchases
  referral history

تحقق من الأمور التالية:

- هل توجد mock data؟
- هل توجد arrays ثابتة مكتوبة داخل الملفات؟
- هل توجد placeholder values؟
- هل توجد useState بقيم ثابتة غير مربوطة بـ API؟
- هل توجد fake stats مثل:
  totalReferrals: 12
  totalRewards: 100
- هل البيانات تأتي من props ثابتة؟
- هل الصفحة تعتمد على بيانات مؤقتة بدل fetch حقيقي؟

إذا وجدت أي static/mock data:
- احذفها أو استبدلها بربط حقيقي بالبيانات الديناميكية

━━━━━━━━━━━━━━━━━━
المرحلة 2 — فحص API integration
━━━━━━━━━━━━━━━━━━

تحقق من:
- ما هي endpoints التي يجب أن ترجع بيانات الإحالة؟
- هل هذه endpoints موجودة فعلاً؟
- هل ترجع بيانات من DB فعلاً؟
- هل frontend يستدعيها؟
- هل الاستدعاء يحصل عند page load؟
- هل يحصل بعد refresh؟
- هل يحصل بعد شراء ناجح؟
- هل هناك React Query / SWR / fetch / axios مستخدم بشكل صحيح؟

إذا كانت الصفحة لا تستدعي backend:
- أصلحها

إذا كانت تستدعي endpoint خاطئ:
- أصلحها

إذا كانت تستدعي endpoint صحيح لكن لا تستخدم response:
- أصلح ذلك

المرحلة 3 — ربط Dashboard بالمصدر الحقيقي
━━━━━━━━━━━━━━━━━━

أريد أن تصبح البيانات التالية ديناميكية 100% من قاعدة البيانات:

في لوحة المحيل:
- total referrals
- total successful referred purchases
- total rewards earned
- referral code
- referral history
- latest purchases via referral
- reward status

في لوحة الأدمن:
- جميع الإحالات
- إجمالي المكافآت
- عدد الإحالات
- المحيل
- المشتري
- قيمة الشراء
- المكافأة
- الحالة
- التاريخ

كل هذه البيانات يجب أن تأتي من database عبر backend API وليس من static values.

━━━━━━━━━━━━━━━━━━
المرحلة 4 — فحص قاعدة البيانات
━━━━━━━━━━━━━━━━━━

تحقق من الجداول المرتبطة:
- purchases
- referrals
- referral_rewards
- users / wallets / investors
- أي جداول أخرى مرتبطة

حدد:
- هل البيانات موجودة في DB فعلاً؟
- هل query في dashboard تجلبها؟
- هل هناك joins خاطئة؟
- هل هناك mismatch في أسماء الحقول؟
- هل هناك فلترة خاطئة تمنع ظهور النتائج؟

━━━━━━━━━━━━━━━━━━
المرحلة 5 — التحديث بعد الشراء
━━━━━━━━━━━━━━━━━━

أريد أن تتأكد من أن البيانات تتحدث فعلياً بعد العملية:

- بعد نجاح الشراء:
  - هل dashboard يحتاج refresh؟
  - هل هناك re-fetch؟
  - هل هناك invalidation للكاش؟
  - هل state تتحدث؟
  - هل الصفحة تبقى تعرض أرقام قديمة؟

إذا كان الموقع يستخدم:
- React Query
- SWR
- custom fetch
- server components

فأصلح آلية التحديث حتى تظهر البيانات الحقيقية بعد العملية.

━━━━━━━━━━━━━━━━━━
المرحلة 6 — إزالة أي وهم بصري
━━━━━━━━━━━━━━━━━━

إذا كانت الواجهة حالياً تحتوي على:
- fake cards
- placeholder rows
- demo numbers
- sample referral entries

أزلها أو اربطها ببيانات حقيقية.
لا أريد أن يبدو النظام شغالاً بينما هو يعرض فقط قيم ثابتة.

إذا لم توجد بيانات حقيقية:
- اعرض empty state واضح واحترافي
مثل:
  "No referrals yet"
  "No rewards yet"
بدل عرض أرقام مزيفة.

━━━━━━━━━━━━━━━━━━
المرحلة 7 — المطلوب النهائي
━━━━━━━━━━━━━━━━━━

أريد منك في النهاية:

1. تحديد هل الواجهة كانت static أو dynamic
2. تحديد ما الملفات التي كانت تحتوي على بيانات ثابتة
3. تحديد ما الذي تم ربطه فعلياً بـ database
4. إصلاح الواجهة لتصبح ديناميكية بالكامل
5. إصلاح التحديث بعد الشراء أو بعد refresh
6. إعطائي تقريراً واضحاً:
   - ما الذي كان static
   - ما الذي كان مكسوراً
   - ما الذي تم إصلاحه
   - كيف أختبره بنفسي

قواعد صارمة
━━━━━━━━━━━━━━━━━━

- لا تفترض أن dashboard dynamic
- لا تكتفِ بتعديل شكلي
- لا تقل "تم الإصلاح" إلا إذا كانت البيانات تُجلب فعلاً من DB
- لا تترك mock data داخل صفحات production
- إذا لم توجد بيانات، اعرض empty states حقيقية
- أصلح root cause بالكامل
